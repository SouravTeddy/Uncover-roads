Smart Search — #2 + #6

How the search handles "museum" (no location) and natural language queries

The unified search pipeline

Every query goes through this before hitting the API:

Step 1 Keyword extraction — strip filler words, find place types + named locations
"I want some good museum options" → types: [museum]
"hotels near the Eiffel Tower" → types: [hotel], location: "Eiffel Tower"
"museum and parks" → types: [museum, park]
Step 2 Location resolution — if no location found, use current map bbox
"museum" → search within current map viewport
"hotels in Montmartre" → geocode "Montmartre", center search there
Step 3 Parallel API calls — one Nominatim call per type, results merged
"museum and parks" → 2 parallel calls, deduplicated by OSM ID

For keyword extraction — which approach?

A

Client-side dictionary match

A hardcoded map of keywords → place types. Fast, offline, zero cost. Handles: museum, hotel, park, gallery, restaurant, cafe, bar, church, mosque, hospital, pharmacy, ATM, etc.

Pros

  • Instant, no API call
  • Predictable
  • Works offline

Cons

  • Won't understand "somewhere cozy for breakfast"
  • Needs manual upkeep
B

Backend LLM extraction

Send query to a small backend endpoint that uses Claude/GPT to extract types + location. Returns structured JSON: {"types":["museum"],"location":null}

Pros

  • Understands natural language
  • Handles edge cases gracefully

Cons

  • Latency on every keystroke
  • Cost + infra
  • Debounce needed
C

Dictionary first, LLM fallback

Try the dictionary match first (instant). If nothing matches, fire an LLM call once the user stops typing (500ms debounce). Best of both worlds.

Pros

  • Fast for common queries
  • Smart for unusual ones

Cons

  • Slightly more complex
  • LLM cost for edge cases only
